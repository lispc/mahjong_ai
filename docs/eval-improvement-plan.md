# 评估函数改进计划

> 背景：当前 `algo/eval_v2.py` 只考虑向听数、搭子质量、听牌张数，导致 ExpectiMax/MCTS 在 benchmark 中弱于原项目 `algo.py` 的 `eval2`。本计划结合日麻/晋北麻将 AI 常见做法，提出可落地的改进方向。

---

## 当前问题

1. **只奖励速度，不惩罚风险**：不估计弃牌是否会被对手和走。
2. **听牌质量刻画粗糙**：只数“能胡几种牌”，不区分两面/坎张/单骑。
3. **忽略有效进张（ukeire）**：搭子质量是启发式近似，不如直接计算“弃某张后有多少张牌能减少向听数”。
4. **没有对手信息**：对手的弃牌、报听状态完全未利用。
5. **没有报听决策**：晋北麻将“报听后锁手”的灵魂规则未建模。

---

## 改进方向

### 1. Ukeire（有效进张数）

**原理**（参考 Riichi-Trainer、tempai.net、mahjong-tile-efficiency）：
对每一张候选弃牌，计算弃掉后手牌能减少向听数的摸牌集合，并按牌山实际剩余张数加权。

**晋北适配**：
- 不能吃牌，进张只能靠摸和碰/杠，因此 ukeire 比“搭子看起来顺不顺”更直接。
- 应使用 practical ukeire，即扣除已见牌和自己手牌后的剩余张数。

**实现要点**：
```python
ukeire(hand, discard, context) = sum(
    remaining_count(t)
    for t in tiles_that_reduce_shanten(hand - discard)
)
```

**预期效果**：更准确地衡量“打这张牌后多快能听牌/和牌”。

---

### 2. 听牌质量（Wait Quality）

**原理**（参考日麻策略资料）：
听牌时不同待牌和牌概率差异很大：
- 两面待（如 45 万等 36 万）> 坎张（46 万等 5 万）> 边张（12 万等 3 万）> 单骑
- 待幺九/字牌通常比待中张更容易被对手打出
- 待现物/筋牌更安全

**晋北适配**：
- 报听后手牌锁死，待牌质量直接决定能不能和。好型应立即报听，坏型宁可默听改良。

**实现要点**：
```python
wait_score = sum(
    wait_type_quality[t] * remaining_count[t] * safety[t]
    for t in winning_tiles
)
```
`wait_type_quality` 可设：两面=1.0，坎张/边张=0.6，单骑=0.3。

**预期效果**：听牌后选择和牌概率更高的待型，减少听死。

---

### 3. 基础防守（Defense / 弃牌安全度）

**原理**（参考日麻防守理论 + houou-statistics）：
评估不仅要最大化自己手牌价值，还要最小化放炮概率。

| 安全等级 | 判断依据 |
|---|---|
| 很安全 | 自己或对手已经打过的牌（现物 genbutsu） |
| 较安全 | 与现物成“筋”的牌（如 4 万已打，则 1/7 万相对安全） |
| 危险 | 中张（3-7）、宝牌、对手报听后附近牌 |

**晋北适配**：
- 不能吃牌减少了部分进攻路径。
- 一旦某家报听，其打过的牌及周边牌更安全，应偏向 betaori（全弃）。

**实现要点**：
- 在 `ContextV2` 中维护每家弃牌序列。
- 给每张候选弃牌计算 `safety(tile)`。
- 在 expectimax 的弃牌层把目标改为：
  ```
  max 手牌价值 − λ · 放炮风险
  ```

**预期效果**：显著降低点炮率，提升最终排名。

---

### 4. 报听决策

**原理**：
晋北麻将灵魂规则。报听后手牌锁死，只能自摸或打刚摸的牌。

**策略**：
- 好型 + 多待牌 + 局面向晚 → 立即报听，阻止对手推进
- 坏型/待牌危险/局面向早 → 默听改良
- 局势危险且没有改良空间 → 拆听防守

**实现要点**：
- 引擎需要支持“报听”动作及锁手机制。
- AI 层增加 `should_declare_tenpai(hand, context)` 判断。

**预期效果**：充分发挥晋北规则特点，避免听死后只能点炮。

---

### 5. 对手建模（Opponent Modeling）

**原理**（参考 Suphx 的 observable features、houou-statistics 的 discard reading）：
从对手弃牌推断其手牌方向，从而调整安全度和待牌选择。

**可做的轻量推断**：
- 某家长期不打某花色，可能在做该花色或已听该花色。
- 报听者报听前的弃牌可用于排除其待牌范围。
- 每看到一张弃牌，把相关筋牌安全度提高。

**实现要点**：
- 扩展 `ContextV2`，维护 `discards_by_player` 和 `suspicious_suits`。
- 评估函数根据对手信息调整：对手不要的待牌加分，对手可能待的弃牌减分。

**预期效果**：提升防守精度，优化待牌选择。

---

### 6. 融合原项目 `algo.py` 评估

**原理**：
Benchmark 中原 `Baseline`（`algo.py` 的 `eval2`）明显强于 `eval_v2`。其 meld/pair 评估对手牌结构的理解更准确。

**实现要点**：
- 将 `algo.eval0 / eval1 / eval2` 作为额外特征加入 `eval_v2.evaluate`。
- 或直接用 `eval2` 替代 `shanten_fast`，保留 Context/防守扩展。

**预期效果**：低成本、高回报，快速追上 baseline 强度。

---

### 7. 权重调优 / 自对弈学习

**原理**：
特征丰富后，用 CEM/ES/策略梯度等方法学习权重。

**实现要点**：
- 把特征做成线性组合：`score = Σ w_i · f_i(hand, context, discard)`。
- 特征包括：shanten、ukeire、wait quality、defense、safety、algo_eval2 等。
- 用 CEM 或进化策略调权，甚至跑 self-play 迭代。

**预期效果**：让各项特征权重自动适配晋北规则。

---

## 建议优先级

1. **先做 1 + 2（ukeire + wait quality）**：最容易实现，直接针对“速度快、听牌好”。
2. **再做 3（基础防守）**：能立刻降低点炮率，效果可能最明显。
3. **同时做 6（融合 algo.py）**：低成本、高回报。
4. **最后做 4 + 5 + 7**：需要引擎改动或更多工程。
